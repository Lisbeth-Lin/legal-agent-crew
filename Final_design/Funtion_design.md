# 国际公法问答系统功能设计文档

> 文件名：`Funtion_design.md`
> 系统定位：面向国际公法专业学生和教师的可溯源智能问答、案例检索与教学辅助系统。
> 核心目标：基于权威国际公法资料，提供可信问答、准确引用、案例学习、概念解释、案例对比、用户反馈与基础评估能力。

---

## 1. PDF 上传与解析

### 1.1 功能目标

支持管理员或教师上传国际公法相关 PDF 文档，例如 ICJ 判决、WTO 报告、投资仲裁裁决、ITLOS 案件、条约文本、课程讲义等，并将 PDF 转换为可检索、可引用、可向量化的结构化文本。

### 1.2 主要用户

- 管理员
- 教师
- 研究助理

### 1.3 输入

- PDF 文件
- 文档元数据：标题、机构、年份、法律领域、文档类型、语言、来源 URL、课程标签等

### 1.4 处理流程

1. 用户上传 PDF。
2. 系统校验文件格式、大小、重复性和基本元数据。
3. 使用 PDF 解析工具提取文本、页码、标题、脚注和段落结构。
4. 对包含扫描页的 PDF 启用 OCR 兜底解析。
5. 识别段落编号，例如 `para. 12`、`[12]`、`Article 31` 等。
6. 按语义段落或固定长度进行文本切分。
7. 为每个 chunk 生成引用锚点，包括文档 ID、页码、段落号、原文片段。
8. 将解析结果写入文档库，并进入向量化与索引构建流程。

### 1.5 输出

- 结构化文档记录
- 段落级文本 chunk
- 页码与段落号映射
- 引用锚点
- 解析状态：成功、部分成功、失败
- 解析错误日志

### 1.6 数据设计

建议核心表包括：

- `documents`：保存文档基本信息
- `document_pages`：保存页级文本
- `document_chunks`：保存检索用文本块
- `citation_anchors`：保存引用锚点
- `parse_logs`：保存解析过程日志

### 1.7 注意事项

- 国际法文献对页码和段落号要求较高，解析时必须尽量保留原始定位信息。
- 对法律文本不宜只按固定 token 切分，应优先保留条文、段落、案情和法院 reasoning 的完整语义边界。
- 扫描版 PDF 的 OCR 结果需要标记置信度，低置信度内容不应直接作为高可信引用来源。

## 2. 国际公法知识库管理

### 2.1 功能目标

构建可维护、可扩展、可检索的国际公法知识库，统一管理案例、条约、国际组织文件、教材讲义和教师自定义材料。

### 2.2 主要用户

- 管理员
- 教师
- 研究助理

### 2.3 管理对象

- 案例文书：ICJ、WTO、ICC、ITLOS、投资仲裁等
- 国际条约：如《维也纳条约法公约》《联合国宪章》等
- 国际组织文件：UN 文件、ILC 草案、委员会报告等
- 教学资料：课程讲义、阅读材料、案例汇编
- 标注数据：主题标签、争议点、关键段落、案例 brief

### 2.4 主要功能

1. 文档新增、删除、更新。
2. 文档分类与标签管理。
3. 按机构、领域、年份、课程、语言管理资料。
4. 查看 PDF 解析结果和 chunk 切分结果。
5. 手动修正段落号、页码、标题层级和标签。
6. 重新触发解析、向量化和索引构建。
7. 设置文档可见范围，例如公开课程库、教师私有库、班级共享库。

### 2.5 推荐分类体系

一级领域可包括：

- 国际法基础理论
- 条约法
- 国家责任
- 国际争端解决
- 国际经济法 / WTO 法
- 国际投资法
- 国际人权法
- 国际刑法
- 国际海洋法
- 武力使用与国际安全
- 外交与领事关系法
- 国际环境法

### 2.6 输出

- 可检索的结构化知识库
- 可筛选的文档列表
- 每份文档的解析状态、索引状态、更新时间
- 每个主题下的案例和资料集合

### 2.7 注意事项

- 知识库应支持增量更新，避免每次上传新文件都全量重建索引。
- 教师上传的课程材料应与公共国际法资料分开管理，方便按课程隔离检索范围。
- 文档版本需要记录，避免旧版条约、旧版讲义和新版资料混淆。

## 3. 基于 RAG 的问答

### 3.1 功能目标

基于国际公法知识库实现可溯源问答。系统根据用户问题检索相关法律材料，再由大模型结合检索结果生成结构化答案，减少幻觉，提高答案可信度。

### 3.2 主要用户

- 学生
- 教师
- 研究人员

### 3.3 输入

- 用户自然语言问题
- 可选限定条件：法律领域、机构、文档类型、年份、课程范围、语言
- 可选回答格式：简要回答、详细回答、IRAC、案例 brief、对比表、英文回答、中文回答

### 3.4 处理流程

1. 接收用户问题。
2. 对问题进行意图识别：概念解释、案例检索、规则分析、案例对比、考试答题等。
3. 对问题进行查询改写，生成关键词查询和语义查询。
4. 使用混合检索：BM25 关键词检索 + 向量检索。
5. 使用 reranker 对候选 chunk 重新排序。
6. 过滤低相关性内容，保留高置信度上下文。
7. 将检索结果、引用锚点和回答要求输入大模型。
8. 生成结构化答案。
9. 执行引用校验和事实一致性检查。
9.返回答案、引用来源和可继续追问的问题。

### 3.5 输出格式

建议默认输出结构：

```text
1. 直接回答
2. 法律规则
3. 相关案例 / 条约依据
4. 分析说明
5. 引用来源
6. 可继续追问的问题
```

对于案例分析题，可输出 IRAC 结构：

```text
Issue
Rule
Application
Conclusion
Sources
```

### 3.6 数据设计

建议保存：

- `qa_sessions`：问答会话
- `qa_messages`：用户问题与模型回答
- `retrieval_results`：每次回答使用的检索结果
- `answer_citations`：答案中实际引用的来源
- `model_runs`：模型调用记录

### 3.7 注意事项

- 系统必须明确区分“检索依据”和“模型推理”。
- 若知识库中没有足够依据，应回答“当前知识库未检索到充分材料”，而不是编造结论。
- 对争议性国际法问题，应展示不同观点，而不是输出单一绝对结论。

## 4. 答案自动引用来源

### 4.1 功能目标

系统在回答中自动附上可靠来源，帮助学生和教师追溯法律依据。引用应尽量精确到文档、页码、段落号或条文编号。

### 4.2 主要用户

- 学生
- 教师
- 研究人员

### 4.3 输入

- RAG 检索得到的 chunk
- chunk 对应的 citation anchor
- 模型生成的答案内容

### 4.4 处理流程

1. 每个 chunk 在入库时生成 citation anchor。
2. RAG 检索阶段返回 chunk 内容及其来源信息。
3. 答案生成时要求模型只引用检索上下文中的来源。
4. 答案生成后进行引用校验：检查引用是否存在于本次检索结果中。
5. 对缺失来源、虚构段落号、错误页码进行拦截或标记。
6. 前端展示引用卡片，用户可点击查看原文上下文。

### 4.5 推荐引用格式

案例引用：

```text
Nicaragua v. United States, ICJ Reports 1986, para. 190.
```

WTO 报告引用：

```text
Appellate Body Report, US — Shrimp, para. 121.
```

条约引用：

```text
Vienna Convention on the Law of Treaties, Article 31.
```

### 4.6 输出

- 答案正文中的脚注或括号引用
- 引用来源列表
- 可点击原文片段
- 引用置信度

### 4.7 注意事项

- 不允许模型自由生成不存在的引用。
- 引用应优先使用段落号，其次使用页码，再其次使用文档标题和 chunk 编号。
- 如果 PDF 没有段落号，应明确显示页码和原文片段。

## 5. 案例检索与案例摘要

### 5.1 功能目标

帮助用户快速检索国际公法案例，并生成适合法学学习和教学使用的结构化 case brief。

### 5.2 主要用户

- 学生
- 教师
- 研究人员

### 5.3 输入

- 案件名称，例如 `Nicaragua v. United States`
- 关键词，例如 `self-defence`、`effective control`
- 法律领域，例如国家责任、武力使用、WTO 法
- 机构、年份、文档类型等筛选条件

### 5.4 处理流程

1. 用户输入案件名称、关键词或法律问题。
2. 系统在知识库中检索相关案例。
3. 返回案例列表，展示标题、机构、年份、主题、摘要预览。
4. 用户选择某个案例。
5. 系统读取该案例的全文 chunk 和元数据。
6. 生成结构化案例摘要。
7. 标注关键段落和推荐阅读段落。

### 5.5 案例摘要模板

```text
案件名称：
审理机构：
年份：
法律领域：
核心事实：
争议问题：
适用法律：
法院 / 机构推理：
裁判结论：
关键段落：
教学价值：
可延伸讨论问题：
```

### 5.6 输出

- 案例检索结果列表
- 单个案例详情页
- 结构化案例摘要
- 关键段落引用
- 相关案例推荐

### 5.7 注意事项

- 案例摘要必须区分事实、法律问题、法院 reasoning 和结论。
- 对于同名或相似案件，应显示机构和年份，避免混淆。
- WTO 案件应区分 Panel Report 和 Appellate Body Report。

## 6. 法律概念解释

### 6.1 功能目标

为学生提供国际公法核心概念、拉丁法谚、条约术语、判例规则和学说争议的解释，帮助理解课程内容和阅读判决。

### 6.2 主要用户

- 学生
- 教师

### 6.3 输入

- 法律术语：如 `jus cogens`、`opinio juris`、`erga omnes`
- 法律规则：如 `effective control test`
- 条约条文：如 `VCLT Article 31`
- 用户指定解释深度：入门、考试、研究、英文原文辅助阅读

### 6.4 处理流程

1. 判断用户输入是否为法律概念解释请求。
2. 检索知识库中的相关条约、案例、讲义和权威材料。
3. 生成分层解释。
4. 给出典型案例和常见误区。
5. 如果概念存在争议，补充主流观点与争议观点。
6. 给出适合考试或论文使用的表达方式。

### 6.5 输出模板

```text
概念名称：
一句话解释：
详细解释：
法律来源：
典型案例：
与相近概念的区别：
常见误区：
考试 / 论文使用建议：
引用来源：
```

### 6.6 示例概念范围

- 条约法：pacta sunt servanda、reservation、subsequent practice
- 国家责任：attribution、due diligence、necessity、countermeasures
- 习惯国际法：state practice、opinio juris
- 强行法与普遍义务：jus cogens、erga omnes
- 武力使用：armed attack、self-defence、proportionality、necessity
- WTO 法：national treatment、most-favoured-nation treatment、necessity test

### 6.7 注意事项

- 概念解释不能只做百科式回答，应结合条约、案例和课堂使用场景。
- 对容易混淆的概念，应主动给出对比。
- 对尚存争议的问题，应避免绝对化表述。

## 7. 案例对比

### 7.1 功能目标

支持用户比较两个或多个国际公法案例在事实、法律问题、规则、推理路径、结论和后续影响上的异同，服务于课堂讨论、论文写作和考试复习。

### 7.2 主要用户

- 学生
- 教师
- 研究人员

### 7.3 输入

- 两个或多个案例名称
- 对比主题，例如 `self-defence`、`attribution`、`treaty interpretation`
- 可选输出格式：表格、文字分析、课堂讨论版、论文论证版

### 7.4 处理流程

1. 识别待比较案例和比较维度。
2. 检索每个案例的相关段落。
3. 分别提取每个案例的事实、争议问题、适用规则、推理和结论。
4. 按维度生成对比表。
5. 总结共同点、差异点和法律发展脉络。
6. 给出引用来源和延伸阅读建议。

### 7.5 输出模板

| 对比维度 | 案例 A | 案例 B | 说明 |
|---|---|---|---|
| 基本事实 |  |  |  |
| 核心争议 |  |  |  |
| 适用规则 |  |  |  |
| 推理路径 |  |  |  |
| 裁判结论 |  |  |  |
| 后续影响 |  |  |  |

### 7.6 输出内容

- 案例对比表
- 相同点总结
- 差异点总结
- 法律规则演化说明
- 课堂讨论问题或论文写作提示
- 引用来源

### 7.7 注意事项

- 对比不能只比较结论，还应比较法院如何论证。
- 如果两个案例所属制度不同，例如 ICJ 与 WTO，应说明制度背景差异。
- 多案例对比时，应控制输出结构，避免信息堆砌。

## 8. 基础评估日志

### 8.1 功能目标

记录系统运行过程中的关键数据，用于评估问答质量、检索效果、引用准确性、响应速度和系统稳定性，为后续论文实验、系统调优和教学应用评估提供依据。

### 8.2 主要用户
****
- 开发者
- 管理员
- 研究人员
- 教师

### 8.3 记录内容

每次问答建议记录：

- 用户问题
- 问题类型
- 检索关键词
- 检索到的 chunk 列表
- reranker 排序结果
- 最终使用的上下文
- 模型输入 prompt
- 模型输出答案
- 引用来源
- 用户反馈
- 响应时间
- token 消耗
- 错误信息

### 8.4 核心指标

建议基础评估指标包括：

- `retrieval_hit_rate`：检索命中率
- `context_relevance`：上下文相关性
- `answer_faithfulness`：答案忠实度
- `citation_accuracy`：引用准确率
- `answer_relevance`：答案相关性
- `hallucination_rate`：幻觉率
- `latency`：响应时间
- `token_usage`：token 消耗
- `feedback_positive_rate`：用户正反馈率

### 8.5 处理流程

1. 系统为每次问答生成唯一 trace ID。
2. 记录检索、重排、生成、引用校验和反馈全过程。
3. 将日志写入数据库或可观测平台。
4. 定期统计问答质量指标。
5. 对低质量回答进行抽样复核。
6. 输出系统优化报告。

### 8.6 数据设计

建议保存：

- `trace_logs`：完整调用链
- `retrieval_logs`：检索日志
- `generation_logs`：生成日志
- `citation_check_logs`：引用校验日志
- `evaluation_logs`：自动评估结果
- `error_logs`：异常日志

### 8.7 注意事项

- 日志中可能包含用户输入内容，应做好权限管理和隐私保护。
- 评估指标不能只依赖自动分数，应结合教师抽样复核。
- 对法律问答系统而言，引用准确率和答案忠实度应优先于回答流畅度。

## 10. 总体实现建议

本系统的关键不是简单调用大模型回答问题，而是围绕国际公法教学场景构建完整链路：

```text
PDF 资料接入
→ 文档结构化解析
→ 知识库分类管理
→ 混合检索与重排
→ RAG 生成答案
→ 自动引用来源
→ 评估日志
```
其中最重要的三项能力是：
可溯源：每个法律结论都尽量对应案例、条约或文档段落。
可维护：知识库能够持续上传、更新、修正和重新索引。
可评估：系统能够记录检索、生成、引用和反馈全过程，为后续优化提供依据。
